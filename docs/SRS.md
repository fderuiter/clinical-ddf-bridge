# Software Requirements Specification (SRS)

## 1. Scope & Introduction
This document defines the compliant audit and ledger architecture of the Cadence Clinical platform, ensuring traceability and regulatory verification for GxP processes. The system synthesizes upstream Clinical Metadata Management via the Designer Service (Neo4j) with downstream Electronic Data Capture via the Execution Engine (PostgreSQL). By leveraging immutable ledgers, graph-based historical tracking, and comprehensive cryptographic mechanisms, Cadence Clinical guarantees data integrity and simplifies compliance verification.

## 2. Dedicated Compliance Section (21 CFR Part 11 / Annex 11)
Cadence Clinical maps directly to the regulatory mandates established by the FDA (21 CFR Part 11) and EMA (Annex 11) to secure electronic records and signatures.

* **21 CFR § 11.10(a) Validation:** The system maintains a continuous record of configuration paths to ensure reproducible behavior across study states, thereby verifying that the system performs as expected under all historical contexts.
* **21 CFR § 11.10(e) Audit Trails:** All clinical modifications generate automated, secure, and chronological time-stamped audit trails. The system natively prevents the alteration or deletion of audit data.
* **21 CFR § 11.10(g) Authority Checks:** Access to system configurations and clinical modifications is governed by role-based access control tied to user identities and execution privileges, logged on every mutation.
* **21 CFR § 11.10(h) Device Checks:** Terminal and session identities are captured alongside each logical transaction to guarantee the source of data entry.

**Standard Audit Fields:**
Every transactional record throughout the platform includes the following mandatory fields:
* `created_at`: The precise UTC timestamp of the transaction.
* `created_by`: The unique identifier/OIDC subject of the executing user.
* `reason_for_change`: A mandatory, user-supplied or system-generated justification for the modification.
* `version_index`: An incremental integer sequence securing the temporal order of events for a specific clinical entity.

## 3. Application-Layer Event Tracking
Within the EDC Execution Engine, application-layer listeners intercept every state change regarding clinical subject metadata (e.g., subject ID, status, enrollment date, demographic records). Before committing any changes to the PostgreSQL operational tables, these listeners synchronously generate an event object encapsulating the pre-mutation state, the post-mutation state, and the contextual audit fields. This ensures that clinical subject histories capture both the *how* and *why* behind data evolution without degrading transactional throughput.

## 4. Cryptographic Ledger Workflow
To provide non-repudiation and prevent data tampering, the platform utilizes a cryptographic ledger mechanism.
* **Background Execution Cycle:** A background scheduler continuously sweeps new, unsealed audit records generated by the application layer.
* **Cryptographic Hashing & Sealing:** Each block of audit logs is cryptographically hashed using robust algorithms (e.g., SHA-256). Every new block hash includes the hash of the immediately preceding block, creating an unbreakable chain. 
* **Verification Rules:** Background verification routines independently recalculate these chains. Any anomaly—such as a missing entry or a compromised hash—instantly raises high-priority alerts to compliance officers, identifying the exact timestamp and entry of the tampering event.

## 5. Database-Level Auditing Triggers
As a fail-safe mechanism against direct database access or administrative scripts bypassing the application layer, the PostgreSQL Execution Engine leverages immutable database-level triggers.
* **Trigger Architecture:** `AFTER UPDATE` and `AFTER DELETE` triggers are placed on core clinical tables. 
* **Shadow Schema Preservation:** If clinical information is hard-deleted or updated outside the standard application routes, these triggers seamlessly capture the delta and insert it into an isolated, secure shadow schema. This ensures that all clinical subject deletions are preserved in the audit log regardless of the vector of operation.

## 6. Graph-Based Study Versioning
The Designer Service utilizes a Neo4j graph database to track the complex evolution of study configurations, including arms, epochs, visits, branches, and elements.
* **Immutable Timelines:** Instead of overriding existing configuration nodes, modifications result in new versioned nodes. 
* **Historical State Capture:** Directed relationships represent temporal continuity (e.g., `PREVIOUS_VERSION`). Consequently, the historical timeline of a study state is captured as an immutable graph path, enabling users to reconstruct the precise configuration that was active at any point in the clinical trial lifecycle.

## 7. Privacy Controls & Exporting
Regulatory compliance also dictates the strict protection of Personally Identifiable Information (PII) during audit trail exports for external reviews.
* **User Masking:** During the generation of CSV export files for auditors, the system dynamically applies data masking to sensitive demographic or user identifiers.
* **Cryptographic Identifier Hashing:** Rather than exposing plain-text user/subject IDs, the export engine outputs deterministic cryptographic hashes of the identifiers. This structural accountability allows auditors to track modifications made by the same individual or applied to the same subject across the audit ledger without exposing actual identities.
